## Reconciliation notes

1. **The documents are consistent on identity, role, project names, major dates, employers, and core stack.** The extended CV adds role detail; the concise CV compresses it. There is no material date conflict to resolve.
2. **“Decade” versus “since 2015.”** The concise CV calls the TypeScript experience “a decade”; the extended CV says RJ went professional in 2015 and has React since 2016/Next.js since 2018. Because time moves, the README does not repeat a fixed “10 years” number. It says “over a decade” nowhere and instead gives the specific React/Next dates only if needed. This avoids a stale or internally awkward count.
3. **Project typography in extracted text is concatenated.** Visual inspection shows a project label followed by a domain: `11io` + `rj11.io`, `11ai` + `ai.rj11.io`, and `11bench` + `bench.rj11.io`. The README uses the unambiguous labels **11ai** and **11bench**, and uses **rj11.io** as the site/project label rather than guessing at “11io” pronunciation.
4. **Skill categories differ only in grouping.** The concise CV combines UI/data and omits some extended-list items. The README uses a curated subset rather than treating either as an exhaustive certification list.
5. **Some current-project types are confidential by implication.** The CV explicitly calls several data explorers proprietary and names “multiple” dashboards/chats. The README preserves the category without inventing clients, product URLs, outcomes, model providers, data sources, or customer metrics.
6. **Website/link fidelity.** The final uses the exact public routes printed in the CV where present: `rj11.io`, `ai.rj11.io`, `bench.rj11.io`, GitHub `rj11io`, LinkedIn `rj11io`, and the public email. It does not add X, Substack, employer links, or a guessed personal domain.
